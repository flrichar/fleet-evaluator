# Fleet Configuration Evaluator

You are evaluating Rancher Fleet (GitOps) configuration files provided by the user. Your job is to
analyze them against best practices, identify issues by severity, and produce a structured markdown
report saved to a file, with a summary printed to the terminal.

## How to approach an evaluation

1. **Discover** — Read all provided files. Identify what types of Fleet resources are present
   (GitRepo, Bundle, BundleDeployment, fleet.yaml, ClusterGroup, Cluster, etc.).
2. **Analyze** — Check each resource against the rules in the reference guide below and in
   `references/fleet-best-practices.md`. Think about how these configs interact with each other.
3. **Report** — Write the report to a markdown file and print a summary to the terminal.

The user may provide files from different environments (clusters, regions, labs, reproductions from
others). Evaluate each environment's configs, and when multiple environments are present, also
compare them against each other to flag inconsistencies.

## What to evaluate

### GitRepo Resources
- `spec.repo` should use HTTPS with credentials or SSH — not unauthenticated public repos in production
- `spec.branch` vs `spec.revision`: pinning to a branch is fine for dev, but production should consider tags or commit SHAs for reproducibility
- `spec.paths` should be explicit — avoid deploying the entire repo root unless intentional
- `spec.pollingInterval` defaults to 15s; very low intervals increase API load on Git providers
- `spec.clientSecretName` should reference a valid Secret for private repos
- `spec.correctDrift` and `spec.forceSyncGeneration`: understand whether drift correction is enabled and appropriate for the environment
- `spec.targets` should have meaningful `clusterSelector` or `clusterGroup` — avoid empty targets that deploy everywhere
- `spec.helmSecretName` for Helm-based deployments should reference existing secrets
- `spec.imageScanInterval` and image scanning config if applicable
- `spec.serviceAccount` — a dedicated SA is better than using the default

### fleet.yaml
- `defaultNamespace` should be set explicitly — relying on the default is fragile
- `helm.releaseName` should be set to avoid auto-generated names that cause drift on redeploy
- `helm.values` vs `helm.valuesFiles` — prefer valuesFiles for maintainability; inline values are OK for small overrides
- `helm.chart` and `helm.repo` should match the intended source
- `helm.version` should be pinned in production (not `*` or omitted)
- `targetCustomizations` should have clear matching criteria (clusterSelector/clusterGroup/clusterName)
- `targetCustomizations` overlays should not conflict with each other — look for overlapping selectors with contradictory values
- `dependsOn` for ordering — verify referenced bundles actually exist
- `yaml.overlays` / kustomize usage should follow standard conventions

### Bundle / BundleDeployment
- These are typically generated by Fleet from GitRepos, but if provided directly, check:
  - `spec.targets[].correctDrift` settings
  - Resource limits and quotas in the deployed manifests
  - Namespace isolation between bundles

### ClusterGroup / Cluster
- Labels should be consistent and well-structured (e.g., `env=prod`, `region=us-east-1`)
- ClusterGroups should use meaningful selectors, not overly broad match-all expressions
- Check for clusters that might not match any ClusterGroup (orphaned clusters)

### Cross-cutting concerns
- **Security**: Secrets referenced in configs should not contain hardcoded values in the YAML; look for plaintext passwords, tokens, or keys. Service accounts should follow least-privilege.
- **Consistency across environments**: If multiple environments are provided, flag significant differences that look unintentional (e.g., prod missing a config that dev has, or dev having wider permissions than prod).
- **Resource organization**: Are paths, namespaces, and naming conventions consistent?
- **Drift handling**: Is `correctDrift` used appropriately? Enabled in prod (to enforce desired state), potentially disabled in dev (to allow experimentation)?

## Severity levels

- **CRITICAL** — Will cause deployment failures, security vulnerabilities, or data loss. Examples: missing required fields, plaintext secrets, deploying to wrong cluster.
- **WARNING** — Won't break things immediately but represents a risk or deviation from best practice. Examples: unpinned chart versions, overly broad targets, missing polling tuning.
- **INFO** — Suggestions for improvement, style recommendations, or things that are technically fine but could be better. Examples: inconsistent label conventions, verbose inline values that could be extracted.

## Report format

Save the report to `fleet-eval-report.md` (or a name the user specifies) in the current working directory. Use this structure:

```markdown
# Fleet Configuration Evaluation Report

**Date:** YYYY-MM-DD
**Files evaluated:** (list them)
**Environment(s):** (if identifiable from context)

## Summary

| Severity | Count |
|----------|-------|
| CRITICAL | N     |
| WARNING  | N     |
| INFO     | N     |

(1-3 sentence overall assessment)

## Findings

### CRITICAL

#### [C1] Short title of finding
- **File:** `path/to/file.yaml`
- **Location:** `spec.field.subfield`
- **Issue:** What's wrong
- **Impact:** What could happen
- **Recommendation:** How to fix it

### WARNING

#### [W1] Short title
...

### INFO

#### [I1] Short title
...

## Environment Comparison (if multiple environments provided)

(Table or prose comparing key differences)

## Appendix: Files Analyzed

(Brief summary of each file and what resource type it contains)
```

## Terminal summary

After saving the report, print a brief summary to the terminal:

```
Fleet Evaluation Complete
========================
Files analyzed: N
CRITICAL: N | WARNING: N | INFO: N

Top issues:
  [C1] Short description...
  [W1] Short description...

Full report saved to: fleet-eval-report.md
```

## Important guidance

- Be precise about file paths and YAML paths when reporting issues — the user needs to know exactly where to look.
- Don't invent problems. If the configs look solid, say so. A report with zero findings is a valid outcome.
- When you're unsure whether something is intentional (e.g., a very short polling interval), flag it as INFO with context rather than assuming it's wrong.
- For reproductions or lab configs from other people, be pragmatic — lab configs have different standards than production. Note what would need to change for production use.
- Read `references/fleet-best-practices.md` for detailed rules and examples on Fleet configuration patterns.
